# Noah Giesler — Portfolio Site

Static HTML/CSS/vanilla JS portfolio. No database, no build step for the site itself (only for image processing). Deployable as-is to any static host — SiteGround, Netlify, GitHub Pages, S3, anywhere.

## File tree

```
noah-portfolio/
├── index.html                    Home page (one long page: gallery, films, music, contact)
├── films/
│   ├── tuned-in.html             17 film detail pages, one per project
│   ├── forest-of-dreams.html     Each is a standalone static file
│   └── …
├── css/
│   └── style.css                 All styles (home + detail pages)
├── js/
│   ├── films.js                  Film metadata for home-page grid
│   ├── gallery.js                Filter, lightbox, grid rendering
│   └── main.js                   Small utilities
├── data/
│   └── gallery.json              Generated by build_gallery.py — do not hand-edit
├── images/
│   ├── photos/                   ← Put extracted photos here
│   ├── photos-thumbs/            Generated thumbnails (do not populate)
│   ├── stills/                   ← Put extracted film stills here
│   ├── stills-thumbs/            Generated thumbnails (do not populate)
│   └── film-thumbs/              (Optional) custom film card thumbnails
└── scripts/
    ├── build_gallery.py          Image processor — run after adding/removing photos
    ├── generate_film_pages.py    Regenerates film detail HTML from embedded data
    └── requirements.txt
```

## Setup — one-time

```
cd noah-portfolio
pip install -r scripts/requirements.txt
```

If any of Noah's photos are HEIC (iPhone defaults), also:

```
pip install pillow-heif
```

## Populating the gallery

1. Extract `Photography.zip` into `images/photos/` (so the files sit directly in that folder, not in a nested subfolder).
2. Extract `Stills.zip` into `images/stills/`.
3. From the project root, run:
   ```
   python scripts/build_gallery.py
   ```

What the script does:
- Reads every image in `photos/` and `stills/`.
- Writes an optimized full-size version (max 2000px on the long edge, JPEG q=85) back into the source folder. This replaces the originals — it keeps the photo folders themselves at a reasonable size for upload/sync. Originals are deleted once the `.jpg` version is written.
- Writes a small thumbnail (max 600px, JPEG q=80) into `photos-thumbs/` and `stills-thumbs/`.
- Writes `data/gallery.json`, which the site reads at load time.
- Strips EXIF metadata. Honors EXIF orientation before stripping.

If Noah wants to keep the raw originals untouched, open `scripts/build_gallery.py` and set `REPLACE_ORIGINALS = False`. Web versions will then be written into `images/photos-web/` and `images/stills-web/` instead, and the manifest will point to those.

The script is incremental — running it again only reprocesses files whose source has a newer modification time than the outputs.

**Captions.** By default, captions are derived from filenames (underscores and hyphens become spaces). If Noah wants richer captions, open `data/gallery.json` after running the script and edit the `caption` field for each entry. Re-running the script will overwrite those edits, so if this becomes a pattern, we can move captions to a separate file.

## Film thumbnails

For the 14 films hosted on YouTube, card thumbnails are pulled from YouTube at runtime (`img.youtube.com/vi/{id}/maxresdefault.jpg`). If `maxresdefault` doesn't exist for a given video, the JS falls back to `hqdefault`.

For the 3 non-YouTube films (Tuned In, What We May Be, We Gather Here Today), placeholder cards show the film title. When Noah has hand-selected thumbnails, drop them into `images/film-thumbs/` with these names:

- `images/film-thumbs/tuned-in.jpg`
- `images/film-thumbs/what-we-may-be.jpg`
- `images/film-thumbs/we-gather-here-today.jpg`

Then update `js/films.js` — find the relevant film entry and add a line:

```js
customThumb: "images/film-thumbs/tuned-in.jpg",
```

…and in the `filmThumbUrl` helper at the bottom of that same file, check `film.customThumb` before the YouTube fallback. (If you want, I can wire that up now — just didn't want to scaffold dead code for a later pass.)

## Editing film content

Each film's detail page is a standalone HTML file under `films/`. Edit directly — it's plain HTML. The page structure follows the same template across all 17 files.

The home-page film grid (order, titles, categories) is driven by `js/films.js`. If Noah adds a new film later:

1. Copy an existing detail page (e.g., `films/tuned-in.html`) to `films/new-slug.html` and edit.
2. Add a new entry to `FILMS` in `js/films.js`.
3. (Optional) Regenerate all detail pages from the embedded data in `scripts/generate_film_pages.py` — but that's only useful if you edit the Python data instead of the HTML directly.

## Local testing on Windows

Because the site uses `fetch()` to load `data/gallery.json`, opening `index.html` directly via `file://` will break in Chrome (CORS). Run a local web server from the project root:

```
python -m http.server 8000
```

Then visit `http://localhost:8000`.

## Deploying

Upload everything except the `scripts/` folder to your web host. The `scripts/` folder is only used locally for image processing. The rest is pure static — HTML, CSS, JS, JSON, images.

If deploying to SiteGround: the File Manager's bulk upload handles folders fine, but for ~1GB of images, SFTP (FileZilla etc.) will be faster and more reliable.

## Known trade-offs / things worth revisiting

- **Google Fonts dependency.** The site loads Instrument Serif and Newsreader from Google Fonts CDN. Fast, reliable, but adds an external dependency. If Noah wants full self-hosting (e.g., for privacy or offline-grade portability), we can download the font files and serve them from `/fonts/`.
- **YouTube thumbnail quality.** YouTube's `maxresdefault` is 1280×720 — fine at card size but mediocre if cards ever scale up. Noah will want to replace these with hand-selected stills for a polished final version. `images/film-thumbs/` is ready for that.
- **Lightbox is CSS-only structured, JS-behavior.** Plays nicely with keyboard nav (arrow keys, Escape) and click-outside-to-close. There's no swipe support on mobile — if that matters for Noah, say the word and I'll add touch handlers.
- **No back-end.** Contact is mailto. If Noah later wants a proper form (spam-protected), that'll need a service (Formspree, Netlify Forms, or a tiny PHP endpoint on SiteGround).
- **No analytics.** Nothing loaded unless you add it.
